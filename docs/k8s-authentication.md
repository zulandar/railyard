# Kubernetes Authentication Guide

Railyard engine pods run AI agents (Claude, Codex, Gemini, OpenCode, or Copilot) that require provider credentials. This guide covers every supported authentication method and how to configure each one via Helm values.

## Method 1: API Key

**Recommended for:** Enterprise teams, pay-per-token billing, CI/headless environments.

An API key is the simplest approach. The chart maps `auth.apiKey` to the correct environment variable based on your `engine.agentProvider` setting:

| Provider | Environment Variable |
|----------|---------------------|
| `claude` | `ANTHROPIC_API_KEY` |
| `codex` | `OPENAI_API_KEY` |
| `gemini` | `GEMINI_API_KEY` |
| `opencode` | `ANTHROPIC_API_KEY` |
| `copilot` | `GH_TOKEN` (via `auth.copilot.token` or `auth.githubToken`) |

### Steps

1. Create an API key in your provider's console (e.g. [console.anthropic.com](https://console.anthropic.com))
2. Set it in your Helm values:

```yaml
auth:
  method: api_key
  apiKey: "sk-ant-..."

engine:
  agentProvider: claude
```

3. Install or upgrade:

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.apiKey="sk-ant-..."
```

**Cost model:** Pay-per-token. Each API call is billed against the key.

## Method 2: OAuth Token

**Recommended for:** Individual developers and small teams on the Anthropic Max plan.

The OAuth token method uses a token generated by `claude setup-token`. It bills against your Max plan subscription rather than per-token. The chart automatically creates a ConfigMap containing `~/.claude.json` with `hasCompletedOnboarding: true` and mounts it into engine pods at `/home/railyard/.claude.json`.

### Steps

1. Run the setup command locally:

```bash
claude setup-token
```

2. Copy the token that is output.
3. Set it in your Helm values:

```yaml
auth:
  method: oauth_token
  oauthToken: "oat_..."

engine:
  agentProvider: claude
```

4. Install or upgrade:

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.oauthToken="oat_..."
```

**Cost model:** Fixed monthly Max plan fee. No per-token charges.

**Token rotation:** OAuth tokens are long-lived. Regenerate annually by running `claude setup-token` again and updating the Helm value.

## Method 3: Cloud Provider Auth

### AWS Bedrock

```yaml
auth:
  method: bedrock
  bedrock:
    accessKeyID: "AKIA..."
    secretAccessKey: "wJalr..."
    region: us-east-1
```

The chart sets `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and `AWS_REGION` in the auth secret.

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.method=bedrock \
  --set auth.bedrock.accessKeyID="AKIA..." \
  --set auth.bedrock.secretAccessKey="wJalr..." \
  --set auth.bedrock.region=us-east-1
```

### Google Vertex AI

```yaml
auth:
  method: vertex
  vertex:
    projectID: my-gcp-project
    region: us-central1
    credentialsSecret: gcp-sa-key    # existing Secret with 'credentials.json' key
```

The chart sets `CLOUD_ML_REGION` and `GOOGLE_CLOUD_PROJECT` in the auth secret. When `credentialsSecret` is provided, it mounts the service account JSON at `/var/secrets/google/credentials.json` and sets `GOOGLE_APPLICATION_CREDENTIALS` to that path.

Create the credentials secret before installing:

```bash
kubectl create secret generic gcp-sa-key \
  --from-file=credentials.json=./service-account.json
```

Then install:

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.method=vertex \
  --set auth.vertex.projectID=my-gcp-project \
  --set auth.vertex.region=us-central1 \
  --set auth.vertex.credentialsSecret=gcp-sa-key
```

### Azure AI Foundry

```yaml
auth:
  method: foundry
  foundry:
    apiKey: "abc123..."
    endpoint: "https://my-resource.openai.azure.com/"
```

The chart sets `AZURE_API_KEY` and `AZURE_ENDPOINT` in the auth secret.

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.method=foundry \
  --set auth.foundry.apiKey="abc123..." \
  --set auth.foundry.endpoint="https://my-resource.openai.azure.com/"
```

## GitHub PAT (PR Mode)

**Required when:** `require_pr: true` is set in your Railyard config.

When PR mode is enabled, the yardmaster creates draft PRs for completed cars and monitors them for review feedback. This requires the `gh` CLI (pre-installed in the engine image) and a GitHub Personal Access Token with **repo** scope.

The chart sets `GH_TOKEN` in the auth secret. This env var is used by the `gh` CLI for all GitHub API operations.

### Steps

1. Create a fine-grained PAT at [github.com/settings/tokens](https://github.com/settings/tokens?type=beta) with repository permissions for **Contents** (read/write) and **Pull requests** (read/write).

2. Set it in your Helm values:

```yaml
auth:
  githubToken: "ghp_..."
```

3. Install or upgrade:

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.githubToken="ghp_..."
```

**Note:** `auth.githubToken` is independent of the agent auth method. You can combine it with any method (API key, OAuth token, Bedrock, etc.).

**Using an existing secret:** If you set `auth.existingSecret`, add `GH_TOKEN` to your pre-created Secret alongside the agent credentials.

## GitHub Copilot CLI

**Recommended for:** Teams using GitHub Copilot as their AI coding agent.

Copilot CLI authenticates via `GH_TOKEN` (or `GITHUB_TOKEN`) using a GitHub Personal Access Token. The PAT requires the **Copilot Requests** permission (fine-grained token) to access the Copilot API.

The chart keeps `GH_TOKEN` (from `auth.githubToken`) intact for PR operations and Bull, and exposes `auth.copilot.token` as a separate `GITHUB_COPILOT_TOKEN` env var. The copilot provider overrides `GH_TOKEN` only in its own subprocess, so other components always see the repo-scoped token.

This allows simple setups (one PAT for everything) and advanced setups (separate tokens with different scopes).

### Simple setup (shared token)

If your PAT has both **Copilot Requests** and **repo** permissions:

```yaml
auth:
  githubToken: "ghp_..."

engine:
  agentProvider: copilot
```

### Advanced setup (separate tokens)

Use a dedicated Copilot token with only **Copilot Requests** permission, and a separate PAT for PR operations:

```yaml
auth:
  githubToken: "ghp_pr_operations_token"
  copilot:
    token: "ghp_copilot_only_token"

engine:
  agentProvider: copilot
```

### Helm install

```bash
helm upgrade --install railyard ./charts/railyard \
  -f values.yaml \
  --set auth.copilot.token="ghp_..." \
  --set engine.agentProvider=copilot
```

**Cost model:** Copilot is billed through your GitHub Copilot subscription (individual, business, or enterprise).

## Advanced: Vault-based Key Rotation

For dynamic credential rotation (e.g. via HashiCorp Vault), use `auth.apiKeyHelper` instead of a static key. When set, the chart populates the `ANTHROPIC_API_KEY_HELPER` environment variable. The agent runtime calls this helper command each time it needs a key, so credentials can rotate without redeploying pods.

```yaml
auth:
  method: api_key
  apiKeyHelper: "vault read -field=key secret/anthropic"
```

This value is set alongside (not instead of) the auth method. It can be combined with `api_key` or `existingSecret`.

## Comparison Table

| Method | Billing | Rotation | Complexity | Best For |
|--------|---------|----------|------------|----------|
| API Key | Pay-per-token | Manual | Low | Enterprise, CI, headless |
| OAuth Token | Max plan (fixed) | Annual | Low | Small teams, individual devs |
| AWS Bedrock | AWS billing | IAM rotation | Medium | AWS-native orgs |
| Google Vertex AI | GCP billing | SA key rotation | Medium | GCP-native orgs |
| Azure AI Foundry | Azure billing | Portal rotation | Medium | Azure-native orgs |
| Vault helper | Pay-per-token | Automatic | High | Strict rotation policies |
| GitHub Copilot | Copilot subscription | PAT rotation | Low | GitHub-native teams |

## Using Existing Secrets

If you manage secrets externally (e.g. Sealed Secrets, External Secrets Operator, or manual `kubectl create secret`), set `auth.existingSecret` to the name of your pre-created Secret:

```yaml
auth:
  method: api_key
  existingSecret: my-railyard-auth
```

The Secret is mounted via `envFrom` on all engine pods. It must contain the correct keys for the chosen method:

| Method | Required Keys |
|--------|--------------|
| `api_key` (claude/opencode) | `ANTHROPIC_API_KEY` |
| `api_key` (codex) | `OPENAI_API_KEY` |
| `api_key` (gemini) | `GEMINI_API_KEY` |
| `api_key` (copilot) | `GH_TOKEN` |
| `oauth_token` | `CLAUDE_CODE_OAUTH_TOKEN` |
| `bedrock` | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION` |
| `vertex` | `CLOUD_ML_REGION`, `GOOGLE_CLOUD_PROJECT` |
| `foundry` | `AZURE_API_KEY`, `AZURE_ENDPOINT` |

Example of creating a secret manually:

```bash
kubectl create secret generic my-railyard-auth \
  --from-literal=ANTHROPIC_API_KEY="sk-ant-..."
```

When `auth.existingSecret` is set, the chart skips creating its own `<release>-auth` Secret and references yours instead.
